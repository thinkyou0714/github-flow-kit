## 基本の使い方

PR のブランチにチェックアウトした状態で実行するだけです。

```bash
/pr-respond
```

Claude Code が自動的に:
1. GitHub API から未解決コメントを全件取得
2. コメントを 4 種類に分類 (MUST-FIX / ACK / DISCUSS / SKIP)
3. MUST-FIX はファイルを編集してコミット
4. ACK / DISCUSS は返答コメントを投稿

## 4 種類の分類ロジック

| 分類 | 判定基準 | 自動アクション |
|---|---|---|
| MUST-FIX | "fix", "change", "wrong", "should" など | ファイル編集 → commit |
| ACK | "nit:", "optional", "minor" など | 了解コメント返答 |
| DISCUSS | "why did you...?", "what if..." など | 理由説明コメント返答 |
| SKIP | 解決済み・絵文字のみ・"LGTM" | 無視 |

## 実践シナリオ 5 選

### シナリオ 1: 型エラーの指摘

```
@reviewer: この関数の戻り値は string ではなく number であるべき
```

**自動対応**: 該当ファイルの型を修正してコミット。返答コメント「型を number に修正しました」を投稿。

### シナリオ 2: Nit 指摘 (変数名)

```
@reviewer: nit: `tmp` より `tempBuffer` の方が意図が明確では？
```

**自動対応**: ファイルは変更せず、「ご指摘ありがとうございます。次回から気をつけます」と返答。

### シナリオ 3: 設計の質問

```
@reviewer: なぜここで sleep(1000) しているのですか？
```

**自動対応**: コードコンテキストを読んで理由を説明するコメントを投稿。

### シナリオ 4: 大量コメント (10件以上)

`--dry-run` で事前確認してから実行することを推奨します。

```bash
/pr-respond --dry-run
# → 変更予定の一覧を表示 (実際には何もしない)

/pr-respond
# → 確認後に実行
```

### シナリオ 5: 特定の PR を指定

複数ブランチで作業しているとき:

```bash
/pr-respond --pr 142    # PR 番号を直接指定
```

## セキュリティ制約

以下のファイルは絶対に変更されません:
- `**/migrations/**` — DB マイグレーション
- `**/.env*` — 環境変数
- `**/auth/**` — 認証ロジック
- `**/.github/workflows/**` — CI/CD

3 ファイル超 / 50 行超の変更が必要な場合は、実行前に確認を求めます。
