## release-notes — リリースノートを 2 分で作る

### 基本

```bash
# カレントブランチの変更からリリースノートを生成
/release-notes

# バージョンを指定
/release-notes --version v1.2.0

# 対象範囲を指定 (タグ間)
/release-notes --from v1.1.0 --to v1.2.0
```

### 出力形式

ユーザー向けと開発者向けの 2 種類が生成されます:

**ユーザー向け (CHANGELOG.md に追記):**
```markdown
## v1.2.0 (2026-04-22)

### 新機能
- ダークモード対応を追加
- CSV エクスポート機能を追加

### バグ修正
- ログインセッションが 24 時間で切れる問題を修正
```

**開発者向け (GitHub Release ドラフトに追加):**
```markdown
## v1.2.0

### Breaking Changes
- `UserConfig` 型から `deprecated` フィールドを削除

### Internal
- Supabase client を v2.42 にアップグレード
```

## issue-triage — バックログを 5 分で整理する

### 基本

```bash
# 全 open issue を優先度スコアリング
/issue-triage

# ラベルでフィルタ
/issue-triage --label bug

# スコア上位 10 件だけ表示
/issue-triage --top 10
```

### スコアリング式

```
Score = 5 × (6 - Effort) + Urgency

Effort  : 1 (数時間) 〜 5 (数週間)
Urgency : 1 (低) 〜 5 (高)
```

### 出力例

```
Issue #142: ログインページが Safari で崩れる
  Impact: 4 | Effort: 2 | Urgency: 5
  Score: 25 ← 最優先
  Labels: bug, frontend

Issue #98: パフォーマンス最適化
  Impact: 3 | Effort: 4 | Urgency: 2
  Score: 12
  Labels: enhancement
```

`/issue-triage` は結果を `TRIAGE.md` に保存します。スプリント計画のインプットとして使えます。

## repo-tour — 初見リポジトリを 2 分で把握する

### 基本

```bash
# カレントリポジトリを解析
/repo-tour

# 出力先を指定
/repo-tour --output REPO_TOUR.md
```

### 生成される内容

```markdown
# Repository Tour: my-project

## 概要
Node.js + TypeScript の REST API。Supabase でデータ永続化。

## ディレクトリ構造 (主要)
src/
  api/       # Express ルートハンドラ
  services/  # ビジネスロジック
  models/    # Supabase スキーマ型定義

## データフロー
Request → Express Router → Service Layer → Supabase

## 開発を始めるには
1. npm install
2. cp .env.example .env (Supabase URL/KEY を設定)
3. npm run dev
```

Mermaid 図も生成されるので、ドキュメントサイトにそのまま貼れます。

## 4 スキルを組み合わせた週次ワークフロー

| タイミング | スキル | 所要時間 |
|---|---|---|
| PR レビューコメント受信後 | `/pr-respond` | 3 分 |
| リリース作業時 | `/release-notes` | 2 分 |
| 週次スプリント計画前 | `/issue-triage` | 5 分 |
| 新規リポジトリ参画時 | `/repo-tour` | 2 分 |

週あたり合計 3〜4 時間の節約。月換算で 12〜16 時間です。
